# Анализ архитектурных паттернов в проекте MiniTMS

## Введение

В этом документе представлен детальный анализ архитектурных паттернов, используемых в проекте MiniTMS. Проект представляет собой яркий пример применения современных подходов к проектированию программного обеспечения, таких как Domain-Driven Design и Clean Architecture.

## Паттерны проектирования

### 1. Domain-Driven Design (DDD)

#### Сущности (Entities)
Сущности в DDD - это объекты с уникальным идентификатором и жизненным циклом. В проекте MiniTMS они находятся в директории `backend/src/domain/entities/`.

Пример: [User entity](backend/src/domain/entities/user.py:38) имеет уникальный идентификатор `id`, а также бизнес-логику в методах `is_locked()`, `increment_failed_attempts()`, `reset_failed_attempts()`.

Особенности:
- Имеют уникальный идентификатор
- Имеют жизненный цикл
- Содержат бизнес-логику
- Могут изменять свое состояние

#### Значимые объекты (Value Objects)
Значимые объекты - это объекты, которые определяются своими значениями, а не уникальным идентификатором. Они находятся в директории `backend/src/domain/value_objects/`.

Примеры:
- **Coordinates** - географические координаты
- **Dimensions** - габариты транспорта или груза
- **Profitability** - показатели рентабельности
- **Route** - маршрут следования

Особенности:
- Неизменяемы (immutable)
- Определяются через свои значения
- Не имеют уникального идентификатора
- Могут использоваться в разных сущностях

#### Сервисы домена (Domain Services)
Сервисы домена содержат бизнес-логику, которая не принадлежит конкретной сущности. Они находятся в директории `backend/src/domain/services/`.

Примеры:
- **ProfitabilityCalculator** - расчет прибыльности рейса
- **RouteOptimizer** - оптимизация маршрутов
- **PlanValidator** - валидация финансовых планов

Особенности:
- Содержат бизнес-логику
- Обычно используют несколько сущностей
- Не имеют собственного состояния
- Могут использовать внешние зависимости через порты

#### Репозитории (Repositories)
Репозитории обеспечивают абстракцию доступа к данным. Они находятся в директории `backend/src/domain/repositories/`.

Примеры:
- **CargoRepository** - репозиторий грузов
- **OrderRepository** - репозиторий заказов
- **PlanRepository** - репозиторий планов
- **UserRepository** - репозиторий пользователей
- **VehicleRepository** - репозиторий транспорта

Особенности:
- Абстрагируют хранилище данных
- Предоставляют интерфейс для CRUD операций
- Определены на уровне домена
- Реализованы на инфраструктурном уровне

#### Доменные события (Domain Events)
Доменные события сигнализируют о произошедших бизнес-событиях. Они находятся в директории `backend/src/domain/events/`.

Примеры:
- **CargoFound** - найден новый груз
- **OrderCreated** - создан новый заказ
- **PlanExceeded** - превышен план

Особенности:
- Иммутабельны
- Содержат данные о произошедшем событии
- Используются для связи компонентов
- Поддерживают слабую связанность

### 2. Чистая архитектура (Clean Architecture)

#### Принципы

1. **Зависимости направлены внутрь** - внешние слои могут зависеть от внутренних, но не наоборот
2. **Внутренние слои не зависят от внешних** - бизнес-логика изолирована от инфраструктурных деталей
3. **Интерфейсы адаптеров** - внешние компоненты соответствуют внутренним контрактам

#### Слои архитектуры

1. **Слой домена (Domain Layer)**
   - Самый внутренний слой
   - Не зависит ни от чего
   - Содержит бизнес-правила
   - Содержит сущности, сервисы, события и репозитории

2. **Слой приложения (Application Layer)**
   - Зависит от слоя домена
   - Содержит варианты использования
   - Определяет интерфейсы (порты) для внешних зависимостей
   - Не содержит инфраструктурных деталей

3. **Инфраструктурный слой (Infrastructure Layer)**
   - Внешний слой
   - Зависит от внутренних слоев
   - Содержит конкретные реализации
   - Включает API, базы данных, внешние сервисы

### 3. Порты и адаптеры (Ports and Adapters)

#### Порты (Ports)
Порты - это интерфейсы, определенные в прикладном слое. Они находятся в директории `backend/src/application/ports/`.

Примеры:
- **EmailPort** - интерфейс для отправки email
- **GPSPort** - интерфейс для получения GPS-данных
- **MapsPort** - интерфейс для работы с картами
- **NotificationPort** - интерфейс для уведомлений
- **ScrapingPort** - интерфейс для скрапинга
- **SheetsPort** - интерфейс для работы с Google Sheets

#### Адаптеры (Adapters)
Адаптеры - это конкретные реализации портов. Они находятся в соответствующих поддиректориях инфраструктурного слоя.

Примеры:
- **SMTP Email Adapter** - реализация EmailPort через SMTP
- **Google Maps Adapter** - реализация MapsPort через Google Maps API
- **Wialon GPS Adapter** - реализация GPSPort через Wialon API
- **Trans.eu Scraping Adapter** - реализация ScrapingPort через Playwright

### 4. Command Query Responsibility Segregation (CQRS)

Хотя проект не полностью реализует CQRS, в нем присутствуют элементы этого паттерна:

- Разделение DTO для команд (например, CreateVehicleRequest) и запросов (например, VehicleDto)
- Отдельные варианты использования для чтения и записи данных
- Использование разных моделей для чтения и записи в некоторых случаях

### 5. Шина событий (Event Bus)

Проект использует шаблон Event Bus для обработки доменных событий:

- События публикуются при изменении состояния доменных объектов
- Подписчики обрабатывают события асинхронно
- Обеспечивается слабая связанность между компонентами
- Реализуется через инфраструктурный компонент EventBus

## Практические примеры реализации паттернов

### Пример реализации Use Case

Структура варианта использования (Use Case) в проекте:

```
backend/src/application/use_cases/cargo/search_cargos.py
```

Этот файл будет содержать реализацию варианта использования поиска грузов, которая:
- Получает входные данные (DTO)
- Взаимодействует с доменными сервисами и репозиториями
- Возвращает результат в виде DTO
- Не содержит инфраструктурных деталей

### Пример реализации Repository Pattern

Интерфейс репозитория находится в:
```
backend/src/domain/repositories/cargo_repository.py
```

А его конкретная реализация в:
```
backend/src/infrastructure/persistence/sqlalchemy/repositories/cargo_repository_impl.py
```

Это позволяет:
- Изолировать бизнес-логику от деталей реализации хранения данных
- Легко заменять реализации (например, для тестирования)
- Поддерживать чистоту архитектуры

### Пример реализации Value Object

[Coordinates](backend/src/domain/value_objects/coordinates.py:1) может быть реализован как иммутабельный объект, содержащий широту и долготу, с методами для вычисления расстояний между координатами.

## Преимущества используемых паттернов

### 1. Тестируемость
- Благодаря четкому разделению ответственности и использованию интерфейсов, компоненты легко тестируются
- Можно создавать mock-объекты для зависимостей
- Бизнес-логика изолирована от инфраструктурных деталей

### 2. Поддерживаемость
- Четкая структура проекта упрощает понимание кода
- Изменения в одной области не влияют на другие
- Легко добавлять новые функциональные возможности

### 3. Расширяемость
- Возможность легко добавлять новые источники данных (например, другие платформы грузоперевозок)
- Возможность интегрировать новые внешние сервисы через порты
- Возможность масштабировать систему по горизонтали

### 4. Надежность
- Четкое разделение ответственности снижает вероятность ошибок
- Доменные события обеспечивают согласованность данных
- Слабая связанность компонентов повышает устойчивость системы

## Заключение

Проект MiniTMS демонстрирует продуманное применение современных архитектурных паттернов. Архитектура, построенная на принципах DDD и Clean Architecture, обеспечивает высокую поддерживаемость, тестируемость и расширяемость системы. Использование паттернов портов и адаптеров позволяет легко интегрировать внешние сервисы и заменять реализации без изменения бизнес-логики. Такая архитектура особенно важна для сложных бизнес-приложений, таких как система управления транспортной компанией.
