## 4. Ограничения проекта

- Работа только с порталом trans.eu
- Поддержка 4 языков: RU, EN, SK, PL  
- Соответствие GDPR для хранения данных
- Интеграция с существующими GPS-системами компании

## 5. Ключевые бизнес-требования

| ID | Требование | Приоритет |
|----|------------|-----------|
| BR-001 | Система должна автоматически получать грузы с trans.eu | Критический |

| BR-003 | Система должна учитывать текущее местоположение транспорта | Высокий |
| BR-004 | Система должна поддерживать многопользовательский режим | Высокий |

| BR-006 | Система должна обеспечивать email-коммуникацию | Средний |
| BR-007 | Система должна вести учёт в Google Sheets | Высокий |

## 6. Конкурентные преимущества

1. **Автоматизация поиска** - скрапинг Trans.eu вместо ручного мониторинга
2. **Расчет в реальном времени** - учет GPS-позиции для точного расчета порожнего пробега
3. **Цветовая индикация** - мгновенная визуальная оценка рентабельности
4. **Интеграция с Google Sheets** - привычный интерфейс для бухгалтерии
5. **Финансовое планирование** - план/факт анализ с GPS-одометрами

## 7. Технологические решения

### Backend
- **FastAPI** - современный async веб-фреймворк
- **PostgreSQL** - надежная реляционная БД
- **Celery + Redis** - фоновые задачи и очереди
- **Playwright** - headless браузер для скрапинга SPA

### Frontend  
- **React + TypeScript** - типизированный UI
- **Vite** - быстрая сборка
- **Redux Toolkit** - управление состоянием
- **Leaflet.js** - визуализация карт

### Интеграции
- **OpenStreetMap (OSRM)** - маршрутизация и расчет расстояний
- **Nominatim** - геокодирование
- **Google Sheets API** - синхронизация данных
- **GPS-платформы** - Wialon, GPS-Trace, Navixy

|---------|------------------|

| Время расчета 100 грузов | < 5 сек |

| Время отклика API | < 500 мс |

## 9. Риски и митигация

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Изменение структуры Trans.eu | Средняя | Мониторинг, быстрое обновление парсера |
| Блокировка скрапинга | Низкая | Использование headless браузера, ротация IP |
| Проблемы с GPS-интеграцией | Средняя | Поддержка нескольких платформ |
| Превышение лимитов API | Низкая | Кэширование, оптимизация запросов |

## 10. Архитектурные принципы

1. **Clean Architecture** - разделение на слои (Domain, Application, Infrastructure)
2. **Domain-Driven Design** - бизнес-логика в центре
3. **CQRS** (частично) - разделение команд и запросов
4. **Event-Driven** - доменные события для слабой связности
5. **API-First** - RESTful API как основа интеграции
