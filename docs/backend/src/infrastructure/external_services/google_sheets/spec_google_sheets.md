# Спецификация: Google Sheets Integration (spec_google_sheets.md)

## 1. Введение
Модуль обеспечивает двустороннюю или одностороннюю синхронизацию данных о перевозках (сущность `Order`) с Google Таблицами. Это позволяет использовать привычный табличный интерфейс для ведения бухгалтерии и отчетности, сохраняя автоматизацию ввода данных.

**Цель**: Автоматическое заполнение 16 из 24 полей заказа, минимизация ручного ввода и ошибок.

---

## 2. Технические требования

### 2.1. Аутентификация и Доступ
* **Протокол**: OAuth 2.0.
* **Scopes**: `https://www.googleapis.com/auth/spreadsheets` (Чтение и запись).
* **Управление токенами**:
    * Access Token (время жизни 1 час).
    * Refresh Token (бессрочный, хранится защищенно) для автоматического обновления доступа без участия пользователя.
* **Target Spreadsheet**: ID таблицы должен храниться в настройках компании (Settings). Если ID не задан, система предлагает создать новую таблицу.

### 2.2. Триггеры синхронизации
1.  **Создание заказа (Create Order)**: При конвертации "Груза" (`Cargo`) в "Заказ" (`Order`) в БД системы, создается новая строка в Google Sheet.
2.  **Обновление статуса**: При изменении статуса (например, доставлено), обновляются соответствующие ячейки.
3.  **Обратная синхронизация (Optional)**: Периодический опрос таблицы (polling) на предмет изменения ручных полей (например, "Номер инвойса"), чтобы обновить их в БД MiniTMS.

## 3. Структура Таблицы (Data Mapping)

Система должна строго соблюдать структуру из **24 столбцов**.
* **Источник данных**: Сущности `Order`, `Cargo`, `Vehicle`, `VehiclePosition`.

| Столбец | Заголовок (Header) | Тип данных | Источник / Логика заполнения | Формула / Примечание |
|:---:|---|---|---|---|
| **A** | Дата загрузки | Date | **Auto**: `Cargo.loading_date` | |
| **B** | Источник | Text | **Auto**: Constant | Значение по умолчанию: `"trans.eu"` |
| **C** | Заказчик | Text | **Auto**: `Order.customer_name` | Название компании |
| **D** | Номер заявки | Text | **Auto**: `Cargo.external_id` | ID заявки в системе/trans.eu |
| **E** | Сумма в инвойс | Money (€) | **Auto**: `Order.invoice_amount` | Цена груза (Price) |
| **F** | Номер инвойса | Text | **Manual** | Заполняет бухгалтер вручную |
| **G** | Место загрузки | Text | **Auto**: `Cargo.loading_place` | Город + Страна |
| **H** | Место выгрузки | Text | **Auto**: `Cargo.unloading_place` | Город + Страна |
| **I** | Дистанция | Number (km)| **Auto**: `Cargo.distance_trans_eu` | Расстояние по данным биржи |
| **J** | OSM min distances | Number (km)| **Auto**: `Cargo.distance_osm` | Расстояние по OSRM (OpenStreetMap) |
| **K** | Номер СМР | Text | **Manual** | Номер CMR накладной |
| **L** | Дата выгрузки | Date | **Auto**: `Cargo.unloading_date` | |
| **M** | Вес груза | Number (kg)| **Auto**: `Cargo.weight` | |
| **N** | Маршрут OSM | URL | **Auto**: `Cargo.route_polyline` | Ссылка на OpenStreetMap Route |
| **O** | €/км | Number | **Auto (Calc)** | `= E / J` (Сумма / OSM Distance) |
| **P** | Контакт контрагента| Text | **Auto**: `Order.contact_person` | Email или телефон |
| **Q** | Сканы документов туда | URL | **Manual** | Ссылка на Google Drive (вручную) |
| **R** | Оригиналы документов | Bool | **Manual** | Checkbox / Да/Нет |
| **S** | Кол-во дней | Number | **Auto (Calc)** | `= L - A` (Дата выгрузки - Дата загрузки) |
| **T** | По эл почте | Bool | **Manual** | Отправлены ли документы по email |
| **U** | (резерв) | - | - | Столбец зарезервирован |
| **V** | Оригиналы клиенту | Bool | **Manual** | Отправлены ли оригиналы |
| **W** | Дата поступления клиенту почтой | Date | **Manual** | Дата вручения |
| **X** | Дата ЕТР | Date | **Auto**: `Order.dates.etr` | Estimated Time of Pickup |
| **Y** | Дата АТР | Date | **Auto**: `Order.dates.atr` | Actual Time of Pickup |

## 4. Бизнес-логика и Валидация

### 4.1. Первичная инициализация
Если пользователь подключает *пустую* таблицу:
1.  Система должна автоматически создать строку заголовков (Row 1).
2.  Применить форматирование (жирный шрифт, закрепление первой строки).
3.  Установить ширину столбцов по умолчанию.

### 4.2. Правила записи
* **Непрерывность**: Новые заказы всегда добавляются в первую пустую строку снизу.
* **Идемпотентность**: Если заказ с таким `Order ID` (в скрытых метаданных или поиске по столбцу D) уже существует, система должна обновить существующую строку, а не создавать дубликат.
* **Приоритет данных**:
    * В *Автоматических* полях (A, B, C, D, E, G, H, I, J, L, M, N, O, P, S, X, Y) приоритет имеет **Система MiniTMS**. Изменения пользователя в этих ячейках могут быть перезаписаны при следующей синхронизации.
    * В *Ручных* полях (F, K, Q, R, T, V, W) приоритет имеет **Пользователь**. Система никогда не очищает эти поля.

### 4.3. Обработка ошибок
* **Token Expired**: Попробовать обновить токен через Refresh Token. Если не вышло — отправить уведомление администратору о необходимости переподключения.
* **Sheet Not Found**: Если таблица удалена или нет доступа — приостановить очередь синхронизации и уведомить пользователя.
* **Invalid Data**: При ошибке записи конкретного поля (например, текст в числовом поле) логировать ошибку, но пытаться записать остальные поля.
