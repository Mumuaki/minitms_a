# Спецификация: OpenStreetMap Integration (spec_osm.md)

## 1. Введение
Модуль предоставляет интерфейс для взаимодействия с экосистемой OpenStreetMap (OSM). Он используется для геокодирования адресов, полученных с Trans.eu, обратного геокодирования координат GPS и построения маршрутов для точного расчета дистанций через сервис OSRM (Open Source Routing Machine).

**Ключевая роль**: Обеспечивает точность данных "OSM distances", используемых в формуле расчета рентабельности.

---

## 2. Используемые Сервисы

Система использует следующие инструменты на базе OpenStreetMap:

### 2.1. Nominatim API
Преобразование строковых адресов в географические координаты и обратно.
* **Forward Geocoding**:
    * *Вход*: Строка адреса с Trans.eu (напр., `DE 80997 München`).
    * *Выход*: `lat`, `lon` (напр., `48.1351, 11.5820`).
    * *Назначение*: Определение координат точек погрузки/выгрузки для отображения на карте и расчетов.
    * *Особенность*: Требуется корректный `User-Agent` согласно Usage Policy.
* **Reverse Geocoding**:
    * *Вход*: Координаты GPS (`lat`, `lon`).
    * *Выход*: Населенный пункт, Код страны.
    * *Назначение*: Отображение человекочитаемого местоположения ТС в UI.

### 2.2. OSRM API (Open Source Routing Machine)
Высокопроизводительный расчет маршрутов и дистанций.
* *Вход*: Точка А (Координаты ТС), Точка B (Загрузка), Точка C (Выгрузка).
* *Параметры*: `profile=car`, `geometries=polyline`, `overview=full`.
* *Выход*:
    * `distance`: Дистанция в метрах (конвертируется в км).
    * `geometry`: Закодированная строка маршрута для отрисовки на карте.
* *Назначение*: Заполнение полей `distance_osm` и `route_polyline` в сущности `Cargo`.

## 3. Функциональные Методы

### 3.1. `get_coordinates(address: str) -> Coordinates`
* Очищает входную строку от лишних символов.
* Выполняет запрос к Nominatim API.
* Возвращает объект координат.
* **Кэширование**: Результаты должны кэшироваться (Redis) минимум на 30 дней. Ключ кэша: нормализованная строка адреса.

### 3.2. `calculate_trip_distance(origin, loading, unloading) -> TripDetails`
* Выполняет расчет маршрута через OSRM, состоящего из двух сегментов:
    1.  **Порожний (Empty Run)**: `Origin (GPS)` -> `Loading Point`.
    2.  **Груженый (Freight Run)**: `Loading Point` -> `Unloading Point`.
* Возвращает:
    * `total_distance_km`: Сумма дистанций обоих сегментов.
    * `empty_distance_km`: Дистанция первого сегмента.
    * `polyline`: Объединенная линия маршрута для отображения на Frontend.

### 3.3. `generate_map_link(waypoints: List[Coordinates]) -> str`
* Генерирует URL для открытия маршрута на сайте OpenStreetMap.
* Формат: `https://www.openstreetmap.org/directions?engine=fossgis_osrm_car&route={lat1},{lon1};{lat2},{lon2}`.

## 4. Ограничения и Оптимизация

### 4.1. Квоты и Развертывание
* **Публичные API**: Имеют строгие ограничения по количеству запросов в секунду (напр., 1 запрос в сек для Nominatim).
* **Стратегия оптимизации**:
    1.  Активно использовать кэширование геокодера.
    2.  Для предварительного расчета использовать формулу *Haversine* с коэффициентом.
    3.  **Рекомендация**: При росте нагрузки развернуть собственные инстансы Nominatim и OSRM в Docker-контейнерах для снятия лимитов и обеспечения приватности данных.

### 4.2. Обработка ошибок
* **No Results**: Если адрес не найден в Nominatim, использовать дистанцию с биржи Trans.eu как резервную.
* **Timeouts**: Реализовать механизм повторных запросов (retry) с задержкой.
