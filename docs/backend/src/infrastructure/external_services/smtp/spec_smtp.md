# Спецификация: Email Communication Service (spec_smtp.md)

## 1. Введение
Модуль обеспечивает возможность отправки коммерческих предложений контрагентам напрямую из интерфейса системы (карточки груза). Модуль включает в себя управление шаблонами писем и строгий контроль лимитов отправки для предотвращения блокировки почтового аккаунта.

---

## 2. Конфигурация SMTP
Система должна поддерживать настройку SMTP-сервера для каждого пользователя (или общий для компании, в зависимости от роли).

* **Параметры**:
    * `SMTP_HOST` (напр. `smtp.gmail.com`)
    * `SMTP_PORT` (напр. `587` или `465`)
    * `SMTP_USER` (Email адрес)
    * `SMTP_PASSWORD` (App Password)
    * `USE_TLS` / `USE_SSL`

---

## 3. Функциональные требования

### 3.1. Отправка предложения
Пользователь инициирует отправку из списка грузов.
1.  **Выбор груза**: Пользователь нажимает "Отправить email" на строке груза.
2.  **Генерация контента**: Система подставляет данные груза в выбранный шаблон.
3.  **Предпросмотр**: Пользователь видит текст письма и может внести ручные правки.
4.  **Вложения**: Возможность прикрепить стандартный пакет документов (Лицензия, Страховка).

### 3.2. Шаблонизатор
Система должна поддерживать HTML/Text шаблоны с переменными:
* `{company_name}` — Название заказчика.
* `{contact_person}` — Имя контактного лица.
* `{cargo_details}` — Описание груза (напр., "24t, Munich -> Warsaw").
* `{price}` — Цена предложения.
* `{route}` — Маршрут.
* `{my_company_name}` — Название нашей компании (подпись).

---

## 4. Ограничения и Лимиты (Rate Limiting)
**Критически важный раздел**. Нарушение этих правил может привести к блокировке почты.

### 4.1. Правила отправки
Система должна принудительно контролировать поток исходящих писем на уровне Backend:

1.  **Часовой лимит**: Максимум **50 писем в час** на одного пользователя.
2.  **Интервал**: Минимальная пауза между письмами — **30 секунд**.

### 4.2. Механизм контроля (Throttling Logic)
Перед каждой отправкой система проверяет счетчики в Redis/Database:

1.  `CheckHourlyLimit`:
    * Если `count(last_60_min) >= 50` -> **БЛОКИРОВКА**. Выбросить ошибку: *"Лимит превышен. Попробуйте через X минут"*.
    * Если `count(last_60_min) >= 40` -> **ПРЕДУПРЕЖДЕНИЕ** в UI: *"Внимание: Осталось 10 писем в этом часе"*.
2.  `CheckInterval`:
    * Если `now() - last_email_time < 30 sec` -> **ЗАДЕРЖКА** или блокировка кнопки "Отправить".

### 4.3. Блокировка массовой рассылки
Система должна технически запрещать выбор более 1 груза для одновременной отправки ("Select All" -> "Send"), чтобы избежать мгновенного исчерпания лимитов.

---

## 5. История и Логирование
Все отправленные письма должны сохраняться в сущности `EmailHistory`.
* **Поля**:
    * `user_id`
    * `cargo_id` (Ссылка на груз)
    * `recipient_email`
    * `sent_at` (Timestamp)
    * `status` (`SENT`, `FAILED`)
    * `subject`
* **Назначение**: Отображение в истории взаимодействия с грузом, чтобы диспетчер не отправил предложение дважды одному и тому же клиенту.
